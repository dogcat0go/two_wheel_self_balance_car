整体关系

这个文件不是普通的最终 URDF，而是 xacro 模板。 launch 文件会先把它展开成完整 URDF 字符串，再交给 robot_state_publisher 和 Gazebo 使用。

xacro 文件：定义尺寸、质量、link、joint、Gazebo 扩展和 ros2_control 硬件接口。
gazebo.launch.py：调用 xacro.process_file() 展开模型，并启动 Gazebo、robot_state_publisher、spawn_entity 和控制器 spawner。
Gazebo：读取展开后的 URDF，生成物理模型、碰撞模型、IMU 传感器和 ros2_control 仿真硬件。
ros2_control YAML：声明要加载哪些 controller，以及这些 controller 绑定哪些 joint 或 sensor。

简单说：xacro 负责“机器人长什么样、有哪些接口”，Gazebo 插件负责“在仿真里让这些接口动起来或产生数据”，YAML 负责“让 ROS 2 controller_manager 加载对应控制器”。

xacro / URDF 基础语法

根标签

文件以 <robot> 作为根节点，并声明 xmlns:xacro，这样才能使用 xacro:property 和 xacro:macro。 

<robot xmlns:xacro="http://www.ros.org/wiki/xacro"
       name="two_wheel_balance_65mm">

属性 property

xacro:property 类似变量。比如轮子半径、车体尺寸、质量都集中在文件开头，后面用 ${...} 引用。 

<xacro:property name="wheel_radius" value="0.0325"/>
<xacro:property name="wheel_diameter" value="${2.0 * wheel_radius}"/>

表达式

${...} 中可以做简单数学计算。当前文件用它计算轮距的一半、负方向位置、惯量公式等。 

<origin xyz="0 ${wheel_separation / 2.0} 0" rpy="0 0 0"/>
<origin xyz="0 ${-wheel_separation / 2.0} 0" rpy="0 0 0"/>

宏 macro

xacro:macro 用来复用结构。当前文件把盒子惯量、圆柱惯量、轮子 link、电机 link 做成宏，减少重复。 

<xacro:macro name="wheel_link" params="name">
  <link name="${name}">
    ...
  </link>
</xacro:macro>

URDF 核心概念

  • link：机器人刚体，例如 base_link、left_wheel_link、imu_link。
  • joint：连接两个 link 的关节，例如左右轮是 continuous，电池、板子、IMU 是 fixed。
  • visual：显示外观，主要给 RViz/Gazebo 看。
  • collision：碰撞几何，Gazebo 物理引擎用它计算接触。
  • inertial：质量、质心和转动惯量，直接影响仿真动态效果。
  • origin xyz rpy：局部坐标变换，xyz 是平移，rpy 是 roll/pitch/yaw。
  • axis：关节运动轴。这里左右轮的 axis xyz="0 1 0" 表示轮子绕 y 轴转。

当前坐标约定是 x 向前、y 向左、z 向上，base_link 位于两个轮轴的中点。 平衡车类模型对质心高度、轮子摩擦和惯量非常敏感，修改尺寸或质量后最好重新检查仿真稳定性。

机器人结构

当前模型围绕 base_link 构建。base_link 位于轮轴中心，但车体几何和质心通过 body_com_z 向上偏移，因此它更像一个倒立摆系统。

主车体

base_link 是父 link，包含车体盒子的 visual、collision 和 inertial。

body_mass body_com_z box_inertial

左右轮

left_wheel_link 和 right_wheel_link 通过 left_wheel_joint、right_wheel_joint 连接到 base_link。

continuous joint velocity command

固定部件

电机、电池、控制板、IMU 都通过 fixed joint 固定到 base_link。 它们虽然不运动，但会影响质量分布和 TF 树。

fixed joint extra_joints

IMU

imu_link 上不仅有 URDF 几何，还在 Gazebo 扩展标签里挂了一个 type="imu" 的 sensor。

imu_sensor imu_link

为什么车轮直接连接到 base_link

真实机械结构里，车轮通常安装在电机轴或减速箱输出轴上。但当前 URDF 采用的是仿真中的简化建模： left_wheel_joint 和 right_wheel_joint 直接以 base_link 为 parent，电机 link 则通过 fixed joint 固定到 base_link，只表示电机外壳、质量和外观。 

base_link
├── left_wheel_joint  -> left_wheel_link   # continuous，可转动
├── right_wheel_joint -> right_wheel_link  # continuous，可转动
├── left_motor_joint  -> left_motor_link   # fixed，仅表示电机本体
└── right_motor_joint -> right_motor_link  # fixed，仅表示电机本体

这种写法不代表真实装配顺序，而是把“电机输出轴位置”直接定义成 base_link 到 wheel link 的转动关节。好处是 TF 树更简单， ros2_control 可以直接控制左右轮关节，Gazebo 中也少一层固定 link 与转动 joint 的组合。

如果希望模型更贴近真实机械结构，也可以改成 base_link -> motor_link -> wheel_link： 

base_link
└── left_motor_joint fixed
    └── left_motor_link
        └── left_wheel_joint continuous
            └── left_wheel_link

改成真实传动链后，left_wheel_joint / right_wheel_joint 的 origin 就要改为相对于 motor link 的坐标，而不是相对于 base_link 的坐标。如果 motor link 原点正好放在轮轴中心，wheel joint 的 origin 才可以写成 0 0 0。

Gazebo 标签

URDF 标准主要描述机器人结构。Gazebo 需要的仿真专属信息，则通过 <gazebo> 扩展标签添加。

全局 Gazebo 标签

没有 reference 属性的 <gazebo> 是作用于整个机器人模型的扩展。 当前文件在这里加载 libgazebo_ros2_control.so。 

<gazebo>
  <plugin filename="libgazebo_ros2_control.so" name="gazebo_ros2_control">
    <robot_param>robot_description</robot_param>
    <robot_param_node>robot_state_publisher</robot_param_node>
    <parameters>$(find balance_car_description)/config/ros2_control_balance_65mm.yaml</parameters>
  </plugin>
</gazebo>

针对某个 link 的 Gazebo 标签

带 reference="..." 的 <gazebo> 只作用在某个 link 上。 当前文件用它设置材质、轮胎摩擦、接触参数和 IMU sensor。 

<gazebo reference="left_wheel_link">
  <material>Gazebo/Black</material>
  <mu1>1.45</mu1>
  <mu2>0.85</mu2>
  <kp>1000000.0</kp>
  <kd>4.0</kd>
</gazebo>
  • material：Gazebo 中显示用材质，不等同于 URDF 的 <material>。
  • mu1 / mu2：两个方向的摩擦系数，轮地接触效果主要看这里。
  • kp / kd：接触约束的刚度和阻尼，过低可能穿模，过高可能抖动。
  • minDepth / maxVel：接触修正参数，用来限制接触穿透和修正速度。
  • selfCollide：是否启用自身碰撞，当前 base_link 设置为 false。

Gazebo 插件关系

1. gazebo_ros2_control 插件

libgazebo_ros2_control.so 是 Gazebo 和 ROS 2 control 的桥。它读取机器人描述中的 <ros2_control> 块，创建一个仿真硬件系统 gazebo_ros2_control/GazeboSystem。

robot_state_publisher 发布参数 robot_description。
Gazebo 插件 通过 robot_param 和 robot_param_node 找到这份机器人描述。
ros2_control 块 声明左右轮 joint 的 velocity command interface 和 position/velocity state interface。
controller_manager 按 YAML 加载 joint_state_broadcaster、wheel_velocity_controller 和 imu_sensor_broadcaster。

这里的关键对应关系是： left_wheel_joint / right_wheel_joint 必须同时出现在 URDF joint、<ros2_control> joint 和 YAML controller joints 中。

2. IMU sensor 插件

imu_link 的 Gazebo 扩展里定义了一个名为 imu_sensor 的 IMU 传感器，并加载 libgazebo_ros_imu_sensor.so 把 Gazebo 传感器数据发布到 ROS 2。 

<gazebo reference="imu_link">
  <sensor name="imu_sensor" type="imu">
    <update_rate>250</update_rate>
    ...
    <plugin filename="libgazebo_ros_imu_sensor.so" name="imu_plugin">
      <ros>
        <namespace>imu</namespace>
        <remapping>~/out:=data</remapping>
      </ros>
      <frame_name>imu_link</frame_name>
    </plugin>
  </sensor>
</gazebo>
  • sensor name="imu_sensor"：Gazebo 里的传感器名称，也被 ros2_control 的 sensor 块引用。
  • update_rate：IMU 仿真更新频率，当前是 250 Hz。
  • noise：模拟角速度和线加速度噪声。
  • namespace 加 remapping：通常会把输出映射到类似 /imu/data 的话题。
  • frame_name：消息里的坐标系，当前为 imu_link。

当前文件里同时存在 Gazebo IMU ROS 插件和 imu_sensor_broadcaster。前者负责从 Gazebo sensor 直接发布 ROS IMU 数据；后者通过 ros2_control 的 sensor state interface 发布/广播传感器状态。实际使用时要确认最终需要订阅哪个话题，避免控制器读错数据源。

ros2_control 控制链路

<ros2_control> 块定义“Gazebo 里有哪些硬件接口”。YAML 文件定义“controller_manager 要加载哪些控制器，以及这些控制器使用哪些接口”。

xacro 里的硬件接口

<ros2_control name="GazeboSystem" type="system">
  <hardware>
    <plugin>gazebo_ros2_control/GazeboSystem</plugin>
  </hardware>

  <joint name="left_wheel_joint">
    <command_interface name="velocity">
      <param name="min">${-wheel_velocity_limit}</param>
      <param name="max">${wheel_velocity_limit}</param>
    </command_interface>
    <state_interface name="position"/>
    <state_interface name="velocity"/>
  </joint>
</ros2_control>

YAML 里的控制器

controller_manager:
  ros__parameters:
    update_rate: 200
    joint_state_broadcaster:
      type: joint_state_broadcaster/JointStateBroadcaster
    wheel_velocity_controller:
      type: velocity_controllers/JointGroupVelocityController
    imu_sensor_broadcaster:
      type: imu_sensor_broadcaster/IMUSensorBroadcaster

joint_state_broadcaster

发布关节状态。YAML 中的 extra_joints 把 fixed joints 也放进关节状态输出，便于调试和显示完整结构。

wheel_velocity_controller

类型是 JointGroupVelocityController，控制 left_wheel_joint 和 right_wheel_joint 的 velocity command interface。

imu_sensor_broadcaster

绑定 sensor_name: imu_sensor 和 frame_id: imu_link，名称需要和 xacro 中的 sensor 定义一致。

如果改了 joint 名、sensor 名或 controller 名，需要同步修改三个地方： xacro 的 URDF joint/sensor、xacro 的 <ros2_control> 块、以及 config/ros2_control_balance_65mm.yaml。

launch 启动流程

launch/gazebo.launch.py 把模型和插件串起来。它主要做下面几件事：

  1. 定位包路径和 xacro 文件：get_package_share_directory("balance_car_description")。
  2. 调用 xacro.process_file() 把 xacro 展开成 URDF XML。
  3. 启动 Gazebo，并加载 worlds/empty.world。
  4. 启动 robot_state_publisher，把 robot_description 作为参数提供给其他节点和插件。
  5. 用 gazebo_ros/spawn_entity.py 把机器人实体生成到 Gazebo 世界中。
  6. 用 controller_manager/spawner 依次加载 joint state、轮速控制器和 IMU broadcaster。

这意味着 Gazebo 插件不是独立工作的。它依赖 launch 中的 robot_state_publisher 参数，也依赖 YAML 中声明的 controller 配置。 

ros2 launch balance_car_description gazebo.launch.py

修改检查清单

修改外观尺寸

  • 改 property，例如 body_length、wheel_radius。
  • 确认 visual 和 collision 是否仍然一致。
  • 必要时重新计算惯量或质量。

修改轮子控制

  • URDF joint 名称要和 <ros2_control> 中一致。
  • YAML 中 wheel_velocity_controller.joints 要同步。
  • 速度限制来自 wheel_velocity_limit。
  • 如果把车轮改为挂在 motor link 下，wheel joint 的 origin 要改成相对于 motor link 的坐标。

修改 IMU

  • Gazebo sensor 名称和 ros2_control sensor 名称保持一致。
  • frame_name 和 frame_id 建议都使用 imu_link。
  • 根据控制算法需要调整 update_rate 和噪声。

修改 Gazebo 物理表现

  • 轮胎打滑优先看 mu1 和 mu2。
  • 接触抖动优先看 kp、kd、maxVel。
  • 平衡车倒得过快或过慢，要检查质量、质心和惯量。
