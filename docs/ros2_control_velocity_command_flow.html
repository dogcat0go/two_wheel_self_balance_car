ROS 2 control 速度命令如何进入 Gazebo

当前运行时图谱

你当前看到的节点说明 Gazebo、robot_state_publisher、controller_manager、 wheel controller、IMU 插件都已经启动。关键节点如下：

/gazebo

Gazebo 仿真进程，负责物理世界、碰撞、关节运动、传感器更新。

仿真世界 物理引擎

/gazebo_ros2_control

由 libgazebo_ros2_control.so 创建的 ROS 2 侧节点。 它把 Gazebo 中的 joint 和 ros2_control 的硬件接口连起来。

Gazebo bridge hardware interface

/controller_manager

ros2_control 的控制器管理器。它按 YAML 加载 controller，并周期性执行 read/update/write。

update_rate 200 Hz 加载 controller

/wheel_velocity_controller

轮子速度控制器，类型是 velocity_controllers/JointGroupVelocityController。 它订阅速度数组，并写入左右轮的 velocity command interface。

速度命令入口 left/right wheel

/robot_state_publisher

持有 robot_description，订阅 /joint_states， 发布 /tf 和 /tf_static。

robot_description TF

/joint_state_broadcaster

从 ros2_control 的 state interface 读取关节位置和速度，并发布 /joint_states。

关节状态 反馈

手动命令入口

让轮子转起来时，你在终端发布的是一个 std_msgs/msg/Float64MultiArray 消息： 

ros2 topic pub /wheel_velocity_controller/commands \
  std_msgs/msg/Float64MultiArray \
  "{data: [5.0, 5.0]}"

这个数组的顺序来自 config/ros2_control_balance_65mm.yaml： 

wheel_velocity_controller:
  ros__parameters:
    joints:
      - left_wheel_joint
      - right_wheel_joint
    command_interfaces:
      - velocity

因此 {data: [5.0, 5.0]} 的含义是：

  • data[0] 写给 left_wheel_joint/velocity。
  • data[1] 写给 right_wheel_joint/velocity。
  • 单位是关节角速度，通常按 rad/s 理解。

这里发的是“轮子关节角速度”，不是底盘线速度 m/s。 如果以后要用 cmd_vel 控制小车，需要再写一层节点，把线速度/角速度换算成左右轮角速度。

命令传递主链路

手动发布速度命令后，主链路可以理解成下面这条路径：

1. 控制台 ros2 topic pub
向 /wheel_velocity_controller/commands 发布 Float64MultiArray，例如 [5.0, 5.0]。
2. /wheel_velocity_controller 接收命令
JointGroupVelocityController 按 YAML 中的 joints 顺序解析数组。
3. controller 写入 command interface
数值被写到 left_wheel_joint/velocity 和 right_wheel_joint/velocity 的 command interface。
4. /controller_manager 周期执行 update
按 YAML 中的 update_rate: 200 运行控制循环， 把 controller 的输出交给硬件接口。
5. gazebo_ros2_control/GazeboSystem 接收写入
GazeboSystem 是 ros2_control 的仿真硬件后端。 它把 velocity command 转换为 Gazebo 关节控制。
6. Gazebo 物理引擎更新轮子
Gazebo 在仿真步进中更新左右轮关节，轮子开始转动，并与地面发生摩擦接触。

从 ROS 2 图上看，你直接发布给的是 /wheel_velocity_controller/commands。 从 Gazebo 内部看，真正让轮子动起来的是 libgazebo_ros2_control.so 加载出的 GazeboSystem 对 Gazebo joint 的写入。

节点职责

/robot_state_publisher 不是轮子控制器

你看到 /robot_state_publisher 订阅 /joint_states， 发布 /tf、/tf_static 和 /robot_description。 它的作用是发布机器人模型和 TF，不负责接收轮子速度命令。 

/robot_state_publisher
  Subscribers:
    /joint_states
  Publishers:
    /robot_description
    /tf
    /tf_static

/controller_manager 是控制循环中心

/controller_manager 管理所有 controller。launch 文件中通过 spawner 启动了三个 controller： 

joint_state_broadcaster
wheel_velocity_controller
imu_sensor_broadcaster

它会按照 update_rate: 200 周期执行控制循环： 

read state from GazeboSystem
update active controllers
write commands to GazeboSystem

/gazebo_ros2_control 是 Gazebo 和 ros2_control 的桥

xacro 中这段告诉 Gazebo 加载桥接插件： 

<gazebo>
  <plugin filename="libgazebo_ros2_control.so" name="gazebo_ros2_control">
    <robot_param>robot_description</robot_param>
    <robot_param_node>robot_state_publisher</robot_param_node>
    <parameters>$(find balance_car_description)/config/ros2_control_balance_65mm.yaml</parameters>
  </plugin>
</gazebo>

xacro 中这段告诉 ros2_control：本机器人使用 Gazebo 提供的虚拟硬件： 

<ros2_control name="GazeboSystem" type="system">
  <hardware>
    <plugin>gazebo_ros2_control/GazeboSystem</plugin>
  </hardware>
</ros2_control>

话题职责

/wheel_velocity_controller/commands

速度命令入口。手动控制台发布，或者以后由你自己的平衡控制节点发布。

Float64MultiArray 写命令

/joint_states

关节状态输出，通常由 joint_state_broadcaster 发布， robot_state_publisher 订阅后转换成 TF。

位置 速度

/dynamic_joint_states

ros2_control 风格的更详细接口状态，适合调试每个 joint 的 state interface。

调试 interface state

/tf 与 /tf_static

TF 树输出。轮子转动后，轮子 link 相对 base_link 的姿态会通过这里体现。

坐标变换 可视化

/imu/data

Gazebo IMU 插件输出的话题，来自 /imu/imu_plugin。 后续做平衡控制时可用它读取车体姿态和角速度。

Gazebo IMU plugin

/imu_sensor_broadcaster/imu

ros2_control 的 IMU broadcaster 输出。它与 /imu/data 都可能提供 IMU 信息，实际控制时要明确选用其中一个数据源。

ros2_control IMU

/robot_description

展开后的机器人描述。Gazebo ros2_control 插件通过 robot_description 理解 joint、sensor 和 ros2_control 接口。

URDF 模型描述

/clock

Gazebo 发布的仿真时间。使用 use_sim_time: true 的节点会跟随这个时间。

仿真时间

状态反馈链路

速度命令进入 Gazebo 只是控制的一半。另一半是 Gazebo 把关节状态和传感器状态反馈回 ROS 2。

1. Gazebo 更新物理状态
左右轮关节位置、速度、车体姿态、IMU 数据在仿真步进中更新。
2. GazeboSystem read()
gazebo_ros2_control/GazeboSystem 从 Gazebo joint/sensor 读取 state。
3. joint_state_broadcaster 发布
关节状态通过 /joint_states 和 /dynamic_joint_states 输出。
4. robot_state_publisher 发布 TF
它订阅 /joint_states，结合 URDF 发布 /tf 和 /tf_static。
5. IMU 话题输出
IMU 数据通过 /imu/data 或 /imu_sensor_broadcaster/imu 被控制节点读取。

后续写平衡控制器时，你自己的节点通常会订阅 IMU 和轮子状态，然后发布 /wheel_velocity_controller/commands。这就形成闭环控制。

调试命令

确认 controller 状态

ros2 control list_controllers

期望看到：

joint_state_broadcaster     active
wheel_velocity_controller   active
imu_sensor_broadcaster      active

确认硬件接口

ros2 control list_hardware_interfaces

重点看左右轮 velocity command interface 是否 available/claimed：

left_wheel_joint/velocity
right_wheel_joint/velocity

发布速度命令

# 前进方向测试
ros2 topic pub /wheel_velocity_controller/commands \
  std_msgs/msg/Float64MultiArray \
  "{data: [5.0, 5.0]}"

# 原地旋转测试
ros2 topic pub /wheel_velocity_controller/commands \
  std_msgs/msg/Float64MultiArray \
  "{data: [5.0, -5.0]}"

# 停止
ros2 topic pub --once /wheel_velocity_controller/commands \
  std_msgs/msg/Float64MultiArray \
  "{data: [0.0, 0.0]}"

观察反馈

ros2 topic echo /joint_states
ros2 topic echo /dynamic_joint_states
ros2 topic echo /imu/data
ros2 topic echo /imu_sensor_broadcaster/imu

查看图结构

ros2 node list
ros2 topic list
ros2 node info /wheel_velocity_controller
ros2 topic info /wheel_velocity_controller/commands

如果发布命令后轮子不动，优先检查三件事： wheel_velocity_controller 是否 active、 /wheel_velocity_controller/commands 类型是否正确、 左右轮 velocity command interface 是否被 controller claimed。
